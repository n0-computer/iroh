# Endpoint Discovery

**Version:** 1.0

Endpoint discovery is the process by which an endpoint learns about its own network situation and selects its home relay. This information is used for address publication and NAT traversal.

## Net Report

A net report assesses the endpoint's network capabilities by probing relay servers. The report contains:

| Field | Type | Description |
|-------|------|-------------|
| udp_v4 | bool | Whether UDP over IPv4 is functional |
| udp_v6 | bool | Whether UDP over IPv6 is functional |
| mapping_varies_by_dest_ipv4 | bool? | Whether IPv4 NAT mapping varies by destination (symmetric NAT) |
| mapping_varies_by_dest_ipv6 | bool? | Whether IPv6 NAT mapping varies by destination |
| preferred_relay | RelayUrl? | Relay server with lowest latency |
| relay_latency | map | Per-relay, per-probe-type latency measurements |
| global_v4 | SocketAddrV4? | Endpoint's public IPv4 address |
| global_v6 | SocketAddrV6? | Endpoint's public IPv6 address |
| captive_portal | bool? | Whether a captive portal is detected |

## Probe Types

The net report is generated by running probes against known relay servers:

### HTTPS Probe

An HTTP latency probe to measure round-trip time to the relay server. This works even in restricted network environments where UDP is blocked.

### QAD IPv4 Probe

A QUIC connection to the relay's QAD service (ALPN `/iroh-qad/0`) over IPv4. This measures latency and discovers the endpoint's public IPv4 address via observed address reports.

### QAD IPv6 Probe

Same as the QAD IPv4 probe but over IPv6. Discovers the endpoint's public IPv6 address.

## Probe Execution

### Initial Report

An initial net report runs the full probe suite against all known relay servers. The overall timeout is 30 seconds.

Probes for each relay are staggered to avoid flooding the network:
- Probes within a relay are run concurrently.
- The first successful probe of each type cancels redundant retries for that type.

### Incremental Report

After the initial report, subsequent reports are incremental — they reuse previous results where appropriate and only re-probe as needed. Incremental reports are triggered by:
- Network interface changes
- Timer-based refresh

### Captive Portal Detection

Captive portal detection is deferred until after UDP probes complete (or fail). If UDP works, captive portal detection is skipped. Otherwise, an HTTP request with a challenge/response is sent to a relay server to detect interception.

## Home Relay Selection

The home relay is the relay server with the lowest observed latency across all probe types.

### Selection Algorithm

1. For each relay server, collect latency measurements from all successful probes (HTTPS, QAD IPv4, QAD IPv6).
2. For each relay, take the minimum latency across all probe types.
3. Select the relay with the lowest minimum latency as the preferred relay.
4. If the preferred relay differs from the current home relay, switch to the new relay.

### Home Relay Persistence

The connection to the home relay MUST be maintained persistently:
- It MUST NOT be closed due to inactivity.
- If the connection drops, the endpoint MUST reconnect automatically.
- Keepalive pings (every 15 seconds) ensure the connection remains alive.

### Home Relay Updates

When the home relay changes:
1. Establish a connection to the new home relay.
2. Update the endpoint's published addressing information.
3. Maintain connections to previous relays for ongoing sessions.

## Address Publication

Endpoints publish their addressing information so that remote endpoints can find them.

### Publication Flow

1. The endpoint monitors its own addressing information:
   - Home relay URL
   - Direct IP addresses (if address filter permits)
2. When addressing information changes, the endpoint publishes a new Pkarr signed packet (see [Addressing](addressing.md)).
3. Even if unchanged, the endpoint republishes every 5 minutes to keep records fresh.

### DNS Staggering

When resolving a remote endpoint's address via DNS, queries are staggered with increasing delays to balance speed and reliability:

| Query | Delay from Start |
|-------|-----------------|
| 1st | 0ms |
| 2nd | 200ms |
| 3rd | 300ms |
| 4th | 600ms |
| 5th | 1,000ms |
| 6th | 2,000ms |
| 7th | 3,000ms |

Each individual query has a 3-second timeout. The overall lookup aborts after 6 seconds. The first successful response is used immediately; subsequent queries are cancelled.

## Network Change Detection

Endpoints MUST monitor network interface changes and respond accordingly:

### Major Changes

A major network change (e.g., new interface, IP address change) triggers:
1. A new net report (incremental).
2. Immediate holepunching attempts on all active connections.
3. Home relay re-evaluation.

### Minor Changes

A minor network change (e.g., interface metric change) triggers:
1. Relay connection health checks.
2. Path status re-evaluation.
